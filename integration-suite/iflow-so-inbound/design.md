# IFL_SO_INBOUND Design

## Purpose

IFL_SO_INBOUND is the synchronous inbound iFlow behind SAP API Management for the Week 2 inbound Sales Order POC. It receives a SAP Sales Order API JSON payload, validates headers and required SAP fields, enriches the exchange, publishes the message to JMS_SO_INBOUND, and returns an HTTP 202 ACK.

This iFlow does not call S/4HANA. IFL_SO_ORCHESTRATION consumes JMS_SO_INBOUND later and posts to the standard SAP Sales Order API.

Approved scope remains APIM + SAP Integration Suite + JMS + standard SAP Sales Order API only. No CAP, PostgreSQL, Event Mesh, UI, or additional service is introduced.

## Runtime Path

The caller posts a SAP Sales Order JSON file to APIM. APIM handles authentication, authorization, rate limiting, analytics, and consumer separation. APIM then forwards the request to IFL_SO_INBOUND. The iFlow validates and enriches the message, publishes it to JMS_SO_INBOUND, and returns an ACK. The downstream IFL_SO_ORCHESTRATION iFlow is responsible for consuming the queue and calling the standard SAP Sales Order API.

## HTTPS Sender Adapter

| Setting | Value |
| --- | --- |
| Adapter | HTTPS Sender |
| Method | POST |
| Resource path | /sales-orders |
| Content type | application/json |
| Access pattern | Called by SAP API Management only |
| Response pattern | Return synchronous validation errors or HTTP 202 after JMS publish |

The HTTPS sender endpoint should not be exposed directly to external consumers.

## Required Headers

| Header | Required | Purpose |
| --- | --- | --- |
| Idempotency-Key | Yes | Authoritative duplicate prevention key |
| X-Consumer-ID | Yes | APIM consumer/application identifier |
| X-Correlation-ID | No | Reused when supplied and generated by the iFlow when missing |
| Content-Type | Yes | Must be application/json |

Header validation fails before JMS publish when Idempotency-Key, X-Consumer-ID, or JSON content type is missing or blank.

## Exchange Properties

| Property | Source | Usage |
| --- | --- | --- |
| correlationId | X-Correlation-ID or generated UUID | ACK, JMS metadata, MPL custom field, and error response |
| idempotencyKey | Idempotency-Key | JMS metadata and downstream duplicate handling |
| consumerId | X-Consumer-ID | Monitoring and consumer traceability |
| inboundReceivedAt | Runtime timestamp | Monitoring |
| purchaseOrderByCustomer | Payload field PurchaseOrderByCustomer | Monitoring and downstream trace |
| soldToParty | Payload field SoldToParty | Monitoring and validation trace |
| itemCount | Count of to_Item.results | Monitoring and validation trace |
| validationStatus | Validation scripts | SUCCESS or FAILED |
| jmsQueueName | Constant | JMS_SO_INBOUND |

## Content Modifiers

CM_SetInitialProperties sets correlationId, idempotencyKey, consumerId, inboundReceivedAt, and jmsQueueName. If X-Correlation-ID is absent, it generates a UUID. It must not modify the SAP Sales Order payload body.

CM_SetAckResponse is used only after successful JMS publish. It sets HTTP status 202, Content-Type application/json, and X-Correlation-ID. The response body contains status ACCEPTED, correlationId, and message Sales order accepted for asynchronous processing.

CM_SetErrorResponse is used by validation and technical exception paths. The response body contains status, correlationId, errorCode, and message. Validation errors do not publish to JMS.

## Validation Steps

Header validation checks that Idempotency-Key exists, X-Consumer-ID exists, Content-Type is application/json, and X-Correlation-ID is preserved or generated as correlationId.

JSON parse validation parses the body as JSON. Malformed JSON returns HTTP 400 and does not publish to JMS.

Header-level payload validation requires these SAP fields: SalesOrderType, SalesOrganization, DistributionChannel, OrganizationDivision, SoldToParty, PurchaseOrderByCustomer, CustomerPurchaseOrderType, CustomerPurchaseOrderDate, SalesOrderDate, PricingDate, RequestedDeliveryDate, and to_Item.results. Supported optional fields include IncotermsClassification and to_Text.results.

Item-level validation checks every to_Item.results entry for UnderlyingPurchaseOrderItem, Material, PricingDate, RequestedQuantity, RequestedQuantityUnit, and to_PricingElement.results.

Pricing validation requires every item to have at least one to_PricingElement.results entry. Each pricing element requires ConditionType, ConditionQuantity, ConditionRateValue, and ConditionCurrency.

Text validation treats to_Text.results as optional. When present, each text entry should contain LongTextID and LongText.

## Groovy Scripts

| Script | Inputs | Outputs | Behavior |
| --- | --- | --- | --- |
| GS_ValidateHeaders.groovy | Idempotency-Key, X-Consumer-ID, X-Correlation-ID, Content-Type | correlationId, idempotencyKey, consumerId, validationStatus | Rejects missing required headers, rejects non-JSON content type, and initializes correlationId |
| GS_ValidateSalesOrderPayload.groovy | Original JSON body | purchaseOrderByCustomer, soldToParty, itemCount, validationStatus SUCCESS | Validates required SAP header fields, item fields, pricing structure, and supported text structure |
| GS_PrepareJmsMessage.groovy | Validated exchange | JMS properties and unchanged body | Keeps the original SAP Sales Order JSON body and sets JMS metadata for downstream orchestration |

Validation script failures must throw concise field-level validation exceptions and must prevent JMS publish. IFL_SO_INBOUND must not map to a custom canonical model.

## JMS Receiver Adapter

| Setting | Value |
| --- | --- |
| Adapter | JMS Receiver |
| Queue | JMS_SO_INBOUND |
| Delivery mode | Persistent |
| Message body | Original SAP Sales Order JSON |
| Correlation ID | correlationId |
| Duplicate reference | idempotencyKey |
| Failure handling | Exception subprocess |

JMS_SO_INBOUND is the handoff to IFL_SO_ORCHESTRATION. This iFlow must not publish to Event Mesh and must not persist data to CAP or PostgreSQL.

## ACK Response

ACK is returned only after successful JMS publish. The HTTP status is 202 Accepted. The response body contains status ACCEPTED, the correlationId, and the message Sales order accepted for asynchronous processing. The ACK means the request was accepted for asynchronous processing; it does not mean a SAP sales order has already been created.

## Exception Subprocess

Validation errors include missing Idempotency-Key, missing X-Consumer-ID, invalid JSON, missing SoldToParty, missing Material, missing RequestedQuantity, or missing to_PricingElement.results. Header and malformed request errors return HTTP 400. Payload schema or field validation errors return HTTP 422. The response contains status REJECTED, correlationId, errorCode VALIDATION_ERROR, and message. No JMS publish occurs.

Technical errors include JMS receiver adapter failure, script runtime exception, or unexpected iFlow runtime error. These return HTTP 500 with status FAILED, correlationId, errorCode TECHNICAL_ERROR, and message. MPL must include correlationId, idempotencyKey, consumerId, and exception summary.

## Monitoring Fields

The iFlow sets MPL custom fields for correlationId, idempotencyKey, consumerId, purchaseOrderByCustomer, soldToParty, itemCount, validationStatus, jmsQueueName, and inboundReceivedAt.

Full payload logging should be avoided by default. Payload logging may be enabled only for controlled POC troubleshooting and should be disabled before production hardening.

## Out of Scope

Out of scope for IFL_SO_INBOUND are direct S/4HANA calls, CAP services, PostgreSQL or custom persistence, Event Mesh publishing, UI or sales rep portal implementation, and any architecture redesign outside APIM + SAP Integration Suite + JMS + standard SAP Sales Order API.
