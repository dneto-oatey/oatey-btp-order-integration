# IFL_SO_INBOUND Design

## Purpose

IFL_SO_INBOUND is the synchronous inbound iFlow behind SAP API Management for the Week 2 inbound Sales Order POC. It receives a SAP Sales Order API JSON payload, validates headers and required SAP fields, enriches the exchange, publishes the message to JMS_SO_INBOUND, and returns an HTTP 202 ACK.

This iFlow does not call S/4HANA. IFL_SO_ORCHESTRATION consumes JMS_SO_INBOUND later and posts to the standard SAP Sales Order API.

Approved scope remains APIM + SAP Integration Suite + JMS + standard SAP Sales Order API only. No CAP, PostgreSQL, Event Mesh, UI, or additional service is introduced.

## HTTPS Sender Adapter

| Setting | Value |
| --- | --- |
| Adapter | HTTPS Sender |
| Method | POST |
| Resource path | /sales-orders |
| Content type | application/json |
| Access pattern | Called by SAP API Management only |
| Response pattern | Return synchronous validation errors or HTTP 202 after JMS publish |

APIM remains responsible only for authentication, authorization, rate limiting, analytics, and consumer separation. The iFlow endpoint should not be exposed directly to external consumers.

## Required Headers

| Header | Required | Purpose |
| --- | --- | --- |
| Idempotency-Key | Yes | Authoritative duplicate prevention key |
| X-Consumer-ID | Yes | APIM consumer/application identifier |
| X-Correlation-ID | No | Reused when supplied; generated by iFlow when missing |
| Content-Type | Yes | Must be application/json |

Validation fails before JMS publish when Idempotency-Key, X-Consumer-ID, or JSON content type is missing or blank.

## Exchange Properties

| Property | Source | Usage |
| --- | --- | --- |
| correlationId | Header or generated UUID | ACK, JMS metadata, MPL custom field, error response |
| idempotencyKey | Idempotency-Key | JMS metadata and downstream duplicate handling |
| consumerId | X-Consumer-ID | Monitoring and consumer traceability |
| inboundReceivedAt | Runtime timestamp | Monitoring |
| purchaseOrderByCustomer | Payload | Monitoring and downstream trace |
| soldToParty | Payload | Monitoring and validation trace |
| itemCount | Payload | Monitoring and validation trace |
| validationStatus | Validation scripts | SUCCESS or FAILED |
| jmsQueueName | Constant | JMS_SO_INBOUND |

## Content Modifiers

### CM_SetInitialProperties

Set correlationId, idempotencyKey, consumerId, inboundReceivedAt, and jmsQueueName. If X-Correlation-ID is absent, generate a UUID. Do not modify the sales order payload body.

### CM_SetAckResponse

Used after successful JMS publish. Set HTTP status 202, Content-Type application/json, and X-Correlation-ID. Response body:

~~~json
{
  "status": "ACCEPTED",
  "correlationId": "\${property.correlationId}",
  "message": "Sales order accepted for asynchronous processing"
}
~~~

### CM_SetErrorResponse

Used by validation and technical exception paths. Response body includes status, correlationId, errorCode, and message. Validation errors do not publish to JMS.

## Validation Steps

### Header Validation

- Idempotency-Key exists and is not blank.
- - X-Consumer-ID exists and is not blank.
  - - Content-Type is application/json.
    - - X-Correlation-ID is preserved when present; otherwise correlationId is generated.
     
      - ### JSON Parse Validation
     
      - Parse the body as JSON. Malformed JSON returns HTTP 400 and does not publish to JMS.
     
      - ### Header-Level Payload Validation
     
      - Required SAP header fields:
     
      - - SalesOrderType
        - - SalesOrganization
          - - DistributionChannel
            - - OrganizationDivision
              - - SoldToParty
                - - PurchaseOrderByCustomer
                  - - CustomerPurchaseOrderType
                    - - CustomerPurchaseOrderDate
                      - - SalesOrderDate
                        - - PricingDate
                          - - RequestedDeliveryDate
                            - - to_Item.results
                             
                              - Supported optional header fields include IncotermsClassification and to_Text.results.
                             
                              - ### Item-Level Payload Validation
                             
                              - For every entry in to_Item.results, validate:
                             
                              - - UnderlyingPurchaseOrderItem
                                - - Material
                                  - - PricingDate
                                    - - RequestedQuantity
                                      - - RequestedQuantityUnit
                                        - - to_PricingElement.results
                                         
                                          - ### Pricing Element Validation
                                         
                                          - Each item must have at least one to_PricingElement.results entry. Each pricing element must include:
                                         
                                          - - ConditionType
                                            - - ConditionQuantity
                                              - - ConditionRateValue
                                                - - ConditionCurrency
                                                 
                                                  - ### Text Validation
                                                 
                                                  - to_Text.results is optional. When present, each entry should include LongTextID and LongText.
                                                 
                                                  - ## Groovy Scripts
                                                 
                                                  - ### GS_ValidateHeaders.groovy
                                                 
                                                  - Inputs: Idempotency-Key, X-Consumer-ID, X-Correlation-ID, Content-Type.
                                                 
                                                  - Outputs: correlationId, idempotencyKey, consumerId, validationStatus.
                                                 
                                                  - Behavior: reject missing required headers, reject non-JSON content type, and initialize correlationId.
                                                 
                                                  - ### GS_ValidateSalesOrderPayload.groovy
                                                 
                                                  - Inputs: original JSON body.
                                                 
                                                  - Checks: required header fields, to_Item.results min one item, required item fields, required pricing structure, and supported text structure.
                                                 
                                                  - Outputs: purchaseOrderByCustomer, soldToParty, itemCount, validationStatus SUCCESS.
                                                 
                                                  - Failure behavior: throw a concise validation exception with the missing or invalid field name. Do not publish to JMS.
                                                 
                                                  - ### GS_PrepareJmsMessage.groovy
                                                 
                                                  - Keep the body as the original SAP Sales Order JSON. Set JMS properties for correlationId, idempotencyKey, consumerId, purchaseOrderByCustomer, soldToParty, and itemCount. Do not map to a custom canonical model in this iFlow.
                                                 
                                                  - ## JMS Receiver Adapter
                                                 
                                                  - | Setting | Value |
                                                  - | --- | --- |
                                                  - | Adapter | JMS Receiver |
                                                  - | Queue | JMS_SO_INBOUND |
                                                  - | Delivery mode | Persistent |
                                                  - | Message body | Original SAP Sales Order JSON |
                                                  - | Correlation ID | correlationId |
                                                  - | Duplicate reference | idempotencyKey |
                                                  - | Failure handling | Exception subprocess |
                                                 
                                                  - JMS_SO_INBOUND is the handoff to IFL_SO_ORCHESTRATION. This iFlow must not publish to Event Mesh and must not persist data to CAP or PostgreSQL.
                                                 
                                                  - ## ACK Response
                                                 
                                                  - ACK is returned only after successful JMS publish.
                                                 
                                                  - HTTP status: 202 Accepted
                                                 
                                                  - ~~~json
                                                    {
                                                      "status": "ACCEPTED",
                                                      "correlationId": "<correlationId>",
                                                      "message": "Sales order accepted for asynchronous processing"
                                                    }
                                                    ~~~

                                                    The ACK means the request was accepted for asynchronous processing. It does not mean a SAP sales order has already been created.

                                                    ## Exception Subprocess

                                                    ### Validation Errors

                                                    Examples: missing Idempotency-Key, missing X-Consumer-ID, invalid JSON, missing SoldToParty, missing Material, missing RequestedQuantity, or missing to_PricingElement.results.

                                                    Response: HTTP 400 for malformed/header errors or HTTP 422 for schema/field validation errors. Return JSON with status REJECTED, correlationId, errorCode VALIDATION_ERROR, and message. No JMS publish.

                                                    ### Technical Errors

                                                    Examples: JMS receiver adapter failure, script runtime exception, or unexpected iFlow runtime error.

                                                    Response: HTTP 500 with status FAILED, correlationId, errorCode TECHNICAL_ERROR, and message. MPL must include correlationId, idempotencyKey, consumerId, and exception summary.

                                                    ## Monitoring Fields

                                                    Set MPL custom fields for correlationId, idempotencyKey, consumerId, purchaseOrderByCustomer, soldToParty, itemCount, validationStatus, jmsQueueName, and inboundReceivedAt.

                                                    Avoid full payload logging by default. Payload logging may be enabled only for controlled POC troubleshooting and should be disabled before production hardening.

                                                    ## Out of Scope

                                                    - Direct S/4HANA calls in IFL_SO_INBOUND.
                                                    - - CAP services.
                                                      - - PostgreSQL or custom persistence.
                                                        - - Event Mesh publishing.
                                                          - - UI or sales rep portal implementation.
                                                            - - Architecture redesign outside APIM + SAP Integration Suite + JMS + standard SAP Sales Order API.
