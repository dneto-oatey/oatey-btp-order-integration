# IFL_SO_INBOUND Manual Build Guide

## Purpose

Manual SAP Integration Suite build guide for the current IFL_SO_INBOUND implementation. Runtime remains APIM -> IFL_SO_INBOUND -> JMS_SO_INBOUND -> IFL_SO_ORCHESTRATION -> standard SAP Sales Order API.

No CAP, PostgreSQL, Event Mesh, UI, RFC, BAPI, custom Z services, S/4HANA call inside IFL_SO_INBOUND, canonical model, or payload persistence outside JMS.

## Current Executable Main Flow

Start / HTTPS Sender -> CM_SetInitialProperties -> CM_SetHeaderValidationContext -> GS_ValidateHeaders -> GS_EnsureCorrelationId -> GS_ExtractMonitoringFields -> CM_SetPayloadValidationStatus -> GS_PrepareJmsMessage -> CM_SetJmsHeaders -> Send to JMS Receiver -> CM_SetAckResponse -> End

No JSON Schema Validation step is present in this executable flow.

## JSON Validation Decision

| Finding | Implementation decision |
| --- | --- |
| Target tenant did not provide native JSON Schema Validator | Do not add JSON Schema Validation to executable iFlow |
| EDI Validator availability | Not a valid substitute for JSON payload validation |
| XML Validator availability | Not a valid substitute for JSON payload validation |
| POC validation approach | Validate payload with Groovy logic in GS_ExtractMonitoringFields.groovy |
| Future option | Add JSON Schema validation later only if available in the target tenant |

## HTTPS Sender

| Setting | Value |
| --- | --- |
| Adapter type | HTTPS Sender |
| Address | /sales-orders |
| Method | POST |
| Content-Type | application/json |
| Authentication | OAuth handled by runtime/APIM path |

## Header Rules

| Header | Rule |
| --- | --- |
| Content-Type | Mandatory; must contain application/json |
| X-Correlation-ID | Optional; CPI generates UUID when missing |
| Idempotency-Key | Optional for POC; store empty value when missing |
| X-Consumer-ID | Optional for POC; store UNKNOWN_CONSUMER when missing |

Future APIM runtime should inject X-Consumer-ID from the authenticated application/API Product. Idempotency-Key may be supplied by the consumer or generated by APIM as fallback.

## JMS Receiver

| Setting | Value |
| --- | --- |
| Pattern | One-way Send to JMS Receiver |
| Queue | JMS_SO_INBOUND |
| Access Type | Non-Exclusive |
| Retention | 7 days |
| Transfer Exchange Properties | Enabled |
| Compression | Enabled if configured in tenant |
| Encryption | Enabled if configured in tenant |

Do not use JMS Request Reply. HTTP 202 is returned only after successful JMS Send. JMS send failure returns HTTP 500 through the local exception subprocess.

## Exception Subprocess

Exception Subprocess -> GS_BuildErrorContext -> CM_SetErrorResponse -> End

Rules: do not publish to JMS, return JSON error response, set Content-Type application/json, set CamelHttpResponseCode from property httpStatus, return 400/422 for validation errors, and return 500 for technical or JMS errors.

## Deploy Checklist

| Check | Expected result |
| --- | --- |
| Main flow sequence | Matches the current executable main flow above |
| JSON Schema Validation | Not present |
| Payload validation | Implemented in Groovy |
| JMS | One-way Send to JMS_SO_INBOUND |
| ACK | HTTP 202 only after JMS Send succeeds |
| Error path | Exception subprocess returns JSON 400/422/500 |
| Body handling | Original SAP Sales Order JSON preserved for JMS |
