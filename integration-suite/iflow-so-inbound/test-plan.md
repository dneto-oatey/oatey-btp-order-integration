# IFL_SO_INBOUND Test Plan

## Scope

Test plan summary for the completed IFL_SO_INBOUND POC. The flow uses Groovy-based validation, header normalization to Exchange Properties, one-way Send to JMS_SO_INBOUND, and HTTP 202 only after successful JMS publication.

## Validated Capabilities

| Capability | Status |
| --- | --- |
| HTTPS Endpoint | Validated |
| OAuth Authentication | Validated |
| Header Validation | Validated |
| Correlation Handling | Validated |
| JMS Publication | Validated |
| 202 ACK Response | Validated |
| Exception Subprocess | Validated |
| Runtime Deployment | Validated |

## Test Matrix

| Scenario | Expected result |
| --- | --- |
| Happy Path - Minimal SAP Sales Order JSON | HTTP 202, response contains correlationId, message published to JMS_SO_INBOUND |
| Missing SoldToParty - Payload Validation | HTTP 422 if Groovy validation enforces SoldToParty; validation-depth dependent if not strict yet |
| Missing Content-Type - Header Validation | HTTP 400 or 415; no JMS publication |
| Missing X-Consumer-ID - POC Fallback | Accepted if payload valid; consumerId defaults to UNKNOWN_CONSUMER |
| Missing X-Correlation-ID - Generated by CPI | HTTP 202; response contains generated correlationId |

## Monitoring Verification

Payload logging is not performed for successful transactions. Success path uses standard MPL, correlationId, and queue monitoring. Error path uses exception subprocess, controlled error responses, and error context logging.
