# IFL_SO_INBOUND Test Plan

## Scope

Test plan for the current IFL_SO_INBOUND executable flow. The iFlow validates Content-Type and SAP Sales Order payload with Groovy, sends valid messages to JMS_SO_INBOUND, and returns HTTP 202 only after successful JMS Send.

No JSON Schema Validation step is present. EDI Validator and XML Validator are not valid substitutes for JSON payload validation.

## Current Executable Main Flow

Start / HTTPS Sender -> CM_SetInitialProperties -> CM_SetHeaderValidationContext -> GS_ValidateHeaders -> GS_EnsureCorrelationId -> GS_ExtractMonitoringFields -> CM_SetPayloadValidationStatus -> GS_PrepareJmsMessage -> CM_SetJmsHeaders -> Send to JMS Receiver -> CM_SetAckResponse -> End

## Header Rules

| Header | POC behavior |
| --- | --- |
| Content-Type | Mandatory; must contain application/json |
| X-Correlation-ID | Optional; CPI generates UUID when missing |
| Idempotency-Key | Optional; empty when missing |
| X-Consumer-ID | Optional; UNKNOWN_CONSUMER when missing |

Future APIM runtime should inject X-Consumer-ID from the authenticated application/API Product. Idempotency-Key may be consumer supplied or generated by APIM as fallback.

## Postman Test Matrix

| ID | Scenario | Expected result | JMS |
| --- | --- | --- | --- |
| INB-P01 | Happy Path - Minimal SAP Sales Order JSON | HTTP 202, response contains correlationId | Message published to JMS_SO_INBOUND |
| INB-V01 | Missing SoldToParty - Payload Validation | HTTP 422 if Groovy validation enforces SoldToParty; validation-depth dependent if not strict yet | No message when rejected |
| INB-H01 | Missing Content-Type - Header Validation | HTTP 400 or 415 | No message |
| INB-P02 | Missing X-Consumer-ID - POC Fallback | Accepted if payload valid; consumerId defaults to UNKNOWN_CONSUMER | Message published |
| INB-P03 | Missing X-Correlation-ID - Generated by CPI | HTTP 202, response contains generated correlationId | Message published |

## JMS Verification

| Assertion | Expected result |
| --- | --- |
| Pattern | One-way Send to JMS Receiver |
| Request Reply | Not used |
| Queue | JMS_SO_INBOUND |
| Access Type | Non-Exclusive |
| Retention | 7 days |
| Transfer Exchange Properties | Enabled |
| Compression | Enabled if configured in tenant |
| Encryption | Enabled if configured in tenant |
| ACK timing | HTTP 202 only after successful JMS Send |
| JMS failure | HTTP 500 through exception subprocess |

## Exception Subprocess Verification

| Assertion | Expected result |
| --- | --- |
| Flow | Exception Subprocess -> GS_BuildErrorContext -> CM_SetErrorResponse -> End |
| JMS publish | Exception subprocess never publishes to JMS |
| Response format | JSON error response |
| Content-Type | application/json |
| Status source | CamelHttpResponseCode from property httpStatus |
| Validation errors | 400 or 422 |
| Technical/JMS errors | 500 |

## Out Of Scope Verification

IFL_SO_INBOUND has no S/4HANA receiver adapter, JSON Schema Validation step, JMS Request Reply, CAP service, PostgreSQL persistence, Event Mesh, UI, RFC, BAPI, custom Z service, custom canonical model, or payload persistence outside JMS.
