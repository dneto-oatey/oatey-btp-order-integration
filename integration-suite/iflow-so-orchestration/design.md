# IFL_SO_ORCHESTRATION Build Specification

## Purpose

IFL_SO_ORCHESTRATION consumes accepted SAP Sales Order JSON messages from JMS_SO_INBOUND, invokes the SAP Standard Sales Order API, classifies the SAP response, triggers callback notification, and routes terminal or retry-exhausted failures to DLQ_SO_INBOUND.

## Approved Runtime

APIM -> IFL_SO_INBOUND -> JMS_SO_INBOUND -> IFL_SO_ORCHESTRATION -> SAP Standard Sales Order API -> Callback -> DLQ

## Executable Flow

JMS Sender JMS_SO_INBOUND -> CM_ReadJmsMetadata -> GS_ValidateConsumedMessage -> GS_PrepareSapSalesOrderRequest -> CM_PrepareSapRequest -> Request Reply to SAP Sales Order API -> GS_HandleSapResponse -> CM_SetSuccessContext -> Optional callback notification -> End

## Error Flow

Exception Subprocess -> CM_SetFailedContext -> Retry / Callback / DLQ decision -> GS_PrepareCallbackPayload when applicable -> GS_PrepareDlqPayload when applicable -> CM_SetDlqContext -> Send to DLQ_SO_INBOUND -> End

## POC Metadata Decisions

| Field | POC behavior | Orchestration behavior |
| --- | --- | --- |
| correlationId | Optional inbound; generated by IFL_SO_INBOUND when missing | Must exist when consumed from JMS_SO_INBOUND |
| consumerId | Optional inbound; fallback UNKNOWN_CONSUMER | Preserve fallback or provided value |
| idempotencyKey | Optional; preserved when present; allowed empty | Missing value is warning only, no DLQ routing |

## Idempotency Strategy

Current POC does not enforce mandatory idempotency because no approved durable idempotency persistence exists in the architecture. IFL_SO_ORCHESTRATION preserves idempotencyKey when present and logs a warning when missing.

Missing idempotencyKey must not route a message to DLQ_SO_INBOUND by itself in the POC. Future production implementation may enforce mandatory idempotency after an approved persistence or SAP-supported duplicate-control mechanism is introduced.

## SAP Business Validation

Business validation is performed by SAP S/4HANA through API_SALES_ORDER_SRV. IFL_SO_ORCHESTRATION must not duplicate customer validation, material validation, pricing validation, partner determination, sales area validation, or sales order business rules.

## Architecture Guardrails

Do not introduce CAP, PostgreSQL, Event Mesh, RFC, BAPI, custom Z APIs, direct custom S/4 APIs, UI, or custom persistence. Success path logging is metadata-only and must not log the full payload.
