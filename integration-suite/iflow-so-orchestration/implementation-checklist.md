# IFL_SO_ORCHESTRATION Implementation Checklist

## Purpose

Implementation-ready CPI checklist for building IFL_SO_ORCHESTRATION. The developer should be able to build the iFlow directly from this checklist without making architecture decisions.

## Approved Runtime

APIM -> IFL_SO_INBOUND -> JMS_SO_INBOUND -> IFL_SO_ORCHESTRATION -> SAP Standard Sales Order API -> Callback -> DLQ

## Architecture Guardrails

- [ ] Use JMS_SO_INBOUND as the only orchestration source queue.
- [ ] Use SAP Standard Sales Order API, API_SALES_ORDER_SRV.
- [ ] Use secure credential handling or SAP destination for SAP API authentication.
- [ ] Preserve correlationId, idempotencyKey, and consumerId from JMS metadata.
- [ ] Do not duplicate SAP business validation in Integration Suite.
- [ ] Do not introduce CAP.
- [ ] Do not introduce PostgreSQL.
- [ ] Do not introduce Event Mesh.
- [ ] Do not introduce RFC.
- [ ] Do not introduce BAPI.
- [ ] Do not introduce custom Z APIs.
- [ ] Do not introduce custom persistence.
- [ ] Do not log payloads on the success path.

## POC Metadata Rules

| Field | Current POC behavior | Orchestration requirement | Future production note |
| --- | --- | --- | --- |
| X-Correlation-ID / correlationId | Optional inbound; generated by IFL_SO_INBOUND when missing | Must exist by orchestration stage | Required operational trace key |
| X-Consumer-ID / consumerId | Optional inbound; UNKNOWN_CONSUMER fallback | Preserve value from JMS | May become mandatory by consumer policy |
| Idempotency-Key / idempotencyKey | Optional; preserved when present; allowed to be empty | Do not route to DLQ when missing; log warning only | May become mandatory with approved persistence |

## CPI Component Build Checklist

| Step | Component | Checklist |
| --- | --- | --- |
| 1 | JMS Sender | Queue `JMS_SO_INBOUND`; consume original SAP Sales Order JSON; preserve JMS headers/properties |
| 2 | CM_ReadJmsMetadata | Capture correlationId, idempotencyKey, consumerId, queue name, retry defaults, and timestamps |
| 3 | GS_ValidateConsumedMessage | Validate technical readiness; do not validate SAP business rules |
| 4 | GS_PrepareSapSalesOrderRequest | Preserve original SAP JSON and prepare API context |
| 5 | CM_PrepareSapRequest | Set HTTP headers/properties for API_SALES_ORDER_SRV |
| 6 | Request Reply | Invoke SAP Standard Sales Order API |
| 7 | SAP Receiver | Use secure credential material or destination; no hard-coded credentials |
| 8 | GS_HandleSapResponse | Classify success, SAP business error, transient error, auth/config error, and technical error |
| 9 | CM_SetSuccessContext | Set success metadata and SAP Sales Order number |
| 10 | Callback Receiver | Send optional SUCCESS or FAILED notification with correlationId |
| 11 | Exception Subprocess | Handle validation, SAP, callback, and technical failures |
| 12 | DLQ Sender | Route exhausted or terminal failures to `DLQ_SO_INBOUND` |

## Completion Criteria

- [ ] Valid JMS message calls API_SALES_ORDER_SRV.
- [ ] SAP success extracts Sales Order number.
- [ ] SAP business errors are not blindly retried.
- [ ] SAP transient 503 and timeout errors are retryable.
- [ ] Retry exhaustion routes to `DLQ_SO_INBOUND`.
- [ ] Missing idempotencyKey is allowed in POC and logged as warning only.
- [ ] Missing correlationId fails orchestration validation because it must exist by this stage.
- [ ] Missing consumerId uses the received fallback value `UNKNOWN_CONSUMER` when provided by inbound.
- [ ] Callback payload includes correlationId for all terminal outcomes.
- [ ] Success path logs metadata only.
